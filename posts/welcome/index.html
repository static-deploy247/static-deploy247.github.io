첫 글: 블로그 시작

2026-02-20

블로그 셋업이 완료되었습니다. 이 글은 TOC(우측 목차)와 내부 링크 이동을 확인할 수 있도록 의도적으로 길게 작성한 샘플입니다.

이 블로그 구조

글은 src/pages/posts/*.md에 추가하면 됩니다. 각 파일은 URL 기준으로 /posts/<파일명>/ 경로로 생성됩니다.

  • src/layouts/PostLayout.astro: 포스트 전용 레이아웃
  • src/layouts/BaseLayout.astro: 공통 head/SEO
  • src/config/seo.ts: 기본 SEO 값
  • .github/workflows/deploy.yml: GitHub Pages 배포

배포 흐름

작성한 Markdown 파일은 npm run build에서 정적 HTML로 변환되고, GitHub Actions가 dist 결과물을 Pages로 업로드합니다.
main 브랜치 기준으로 자동 배포되며, 빌드 실패 시 배포도 중단됩니다.

포스팅 작성 팁

소제목 사용

##, ###를 쓰면 우측 TOC에 자동 반영됩니다.
긴 글일수록 h2는 큰 문맥, h3는 세부 기준으로 나누는 것이 읽기 편합니다.

SEO 항목

frontmatter의 title, description, image, noindex로 글별 SEO를 조정할 수 있습니다.

---
layout: ../../layouts/PostLayout.astro
title: "글 제목"
description: "검색 결과와 공유 카드에 노출될 설명"
image: "/images/post-hero.png"
type: "article"
noindex: false
---

정책 글 템플릿 예시

정책 정리형 글은 보통 “배경 -> 대상 -> 변경점 -> 일정 -> 체크리스트” 순서가 읽기 좋습니다.

이미지 예시

아래는 테스트용으로 추가한 정사각형 샘플 이미지입니다.

1) 배경

이 정책이 왜 등장했는지, 어떤 문제를 해결하려는지 한 단락으로 명확히 요약합니다.
정책 문서는 길고 용어가 많기 때문에, 배경 설명이 먼저 나오지 않으면 독자가 핵심을 놓치기 쉽습니다.

2) 적용 대상

정책의 대상이 개인인지, 사업자인지, 특정 업종인지 먼저 구분하세요.

  • 필수 적용 대상
  • 조건부 적용 대상
  • 적용 제외 대상

3) 핵심 변경점

이전 규정과 달라진 항목을 표 형태나 번호 목록으로 정리하면 이해가 빨라집니다.

  1. 기존 기준
  2. 개정 기준
  3. 실무 영향

4) 시행 일정

일정을 잘못 이해하면 실무 대응이 늦어질 수 있습니다.
발표일, 계도기간, 의무 시행일, 과태료 적용 시점 등을 각각 분리해서 적는 편이 안전합니다.

5) 실무 체크리스트

  • 대상 여부 확인
  • 내부 프로세스 변경 필요 여부
  • 시스템/문서 업데이트 항목
  • 사내 공지/교육 필요 여부
  • 일정별 책임자 지정

긴 본문 테스트 섹션 A

스크롤 테스트를 위해 본문 길이를 늘리는 샘플 문단입니다. 정책 정리 글에서는 문장을 짧게 끊고, 문단마다 하나의 메시지만 전달하는 방식을 추천합니다.
핵심 메시지는 문단 첫 문장에 두고, 예외 사항은 뒤에서 보충하면 독자가 빠르게 훑어보기 좋습니다.

A-1. 용어 정리

용어가 혼재되면 해석 오류가 생깁니다. 문서 초반에 용어 표를 두고, 본문에서 동일한 단어를 일관되게 사용하세요.

A-2. 예외 케이스

예외 케이스는 본문에 섞기보다 별도 소제목으로 분리하는 것이 좋습니다.
예외가 많을수록 본문 집중도가 떨어지기 때문에, 조건별로 묶어 정리하세요.

긴 본문 테스트 섹션 B

두 번째 긴 섹션입니다. 실제 운영에서는 정책 원문 링크, FAQ 링크, 공지 링크를 함께 남겨서 추적 가능성을 높이는 것이 좋습니다.

B-1. 링크 관리

외부 문서 링크는 시간이 지나며 깨질 수 있습니다.
가능하면 문서명, 발행 기관, 공고 번호를 함께 기록해두면 링크 변경 후에도 원문 추적이 쉽습니다.

B-2. 버전 히스토리

정책 글은 개정이 잦습니다. 글 하단에 변경 이력을 두면 독자가 최신 여부를 빠르게 판단할 수 있습니다.

긴 본문 테스트 섹션 C

세 번째 긴 섹션입니다. 이 단락도 스크롤 길이를 확보하기 위한 샘플입니다.
실제 글에서는 반복 문장을 줄이고, 반드시 필요한 근거만 남기는 편이 SEO와 체류시간 모두에 유리합니다.

C-1. 내부 검토 포인트

  • 법무 검토 필요 여부
  • 운영 정책 반영 범위
  • 고객 안내 문구 변경 여부
  • 시행 전 리허설 필요 여부

C-2. 게시 전 점검

제목, 요약, 본문 목차, 링크 유효성, 날짜 표기, 책임자 확인 항목까지 체크하면 품질이 안정됩니다.

마무리

이 샘플 글은 TOC와 스크롤 이동 테스트를 위해 길게 구성했습니다.
원하는 실제 정책 주제로 교체할 때도 h2/h3 구조만 유지하면 동일한 사용자 경험을 얻을 수 있습니다.
